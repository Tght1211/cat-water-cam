# 智能猫咪饮水摄像头 — 设计文档（MVP 优先）

- 日期：2026-06-19
- 状态：已确认，待写实现计划

## 1. 目标与背景

做一个**最低成本**的猫咪饮水摄像头，自动识别猫喝水、记录每天喝水时间与次数、本地网页可看可下载最近若干段喝水视频。预算严格控制在 **¥50 以内**。开发者有软件背景、硬件纯小白，软件实现交给 AI 完成。

家中有一台 **Mac Mini 可 24 小时常开、专用**作服务器。摄像头放置点离 Mac Mini 很近（5 米内可走 USB）。水碗是**流动水**（活水饮水机）。

核心原则：**纯本地、零云端、零第三方 API、不外泄隐私**；**MVP 先从无到有，再逐步完善**。

## 2. 关键决策（已与用户确认）

| 主题 | 决策 |
|---|---|
| 大脑/服务器 | Mac Mini，常开专用，跑全部软件 |

| 预算 | **≤ ¥50** |

## 3. 总体架构

```
普通USB摄像头 ──USB线──▶ Mac Mini（大脑，常开专用）
                              ├─ camera：OpenCV 读摄像头出帧
                              ├─ detector：YOLO认猫 + 水碗区域 + 停留 → 喝水事件
                              ├─ recorder：滚动缓存 → 存2~5秒mp4 → 环形只留最近10段
                              ├─ stats：记录每次喝水时间 → 今日次数（SQLite）
                              ├─ feedback：网页👍/👎标注 → 抽训练帧入永久训练集
                              └─ web：本地局域网网页（实时画面/视频列表下载/今日统计/反馈）
你远程进 Mac Mini ──▶ 浏览器开本地网页查看（不碰路由器、不做穿透）
```

全链路在 Mac Mini 本地，画面与视频不出家门。

## 4. 检测逻辑（分阶段）

**闸门（连续跑，0 训练）**
- YOLO 预训练权重检测画面里的 "cat"。低帧率即可（例如 2–5 fps），Mac Mini 轻松胜任。
- 仅当检测到"猫"且其框与用户框定的**水碗区域**重叠时，才进入判定。
- 明确不使用动静/帧差检测——水碗是流动水，帧差会持续误触发。

**喝水判定**
- **MVP**：猫在水碗区域**连续停留 ≥ N 秒**（可配）→ 判定一次喝水事件；带**冷却时间**（可配）避免同一次连续重复计数。
  - 已知局限：会把"凑过来好奇但没喝"的也计入（暂时多计，可接受）。
  - 附带：自动把白天的**猫头特写帧**存下，作为后续训练素材。
- **升级（阶段 2）**：在闸门后加一个"确认"步骤——小模型判断"有没有伸舌头舔水"。有舌头=真喝水；无舌头=路过/好奇，丢弃。

**已知约束**
- 仅白天/有光时工作；黑灯无红外则无法检测（设计上接受漏记夜间）。
- 因无红外，画面始终彩色，训练数据无需考虑夜间黑白问题。

## 5. 反馈标注 + 增量再训练闭环（active learning）

这是"从 0 到 1 + 越用越准"的核心：

```
闸门(YOLO认猫+水碗区域) ─▶ 当前模型判"喝水?" ─▶ 录片 + 计一次
                                                      │
网页每段视频带按钮: 👍真喝水 / 👎没喝(路过好奇) ◀──────┘
        │
        ▼
   该段关键帧 + 标签 存进"训练数据集"（永久）
        │
        ▼
   攒够一批 / 用户点"重新训练" ─▶ 微调舌头模型 ─▶ 自动替换当前模型 ─▶ 越来越准
```

**设计要点**
1. **视频环形缓存（10 段）与训练数据集分离**：给用户看的视频只留最近 10 段（省空间）；用户标注过的帧 + 标签**单独永久存进训练集**，在视频被挤掉前就抽帧入库，避免历史标注随 10 段轮换丢失。
2. **趁视频还在列表时标注**：每段视频在被新视频挤掉前，用户点过 👍/👎，其训练帧即入库。
3. **"0→1" 冷启动**：第一版模型可以很糙（MVP 阶段甚至只用"猫+停留"占位），随用户每天点反馈、数据累积，阶段 2 起模型一轮轮自动变准。

## 6. 录制与存储

- 维护一个**滚动帧缓存**（最近数秒），事件触发时拼成一段约 **3 秒**（可配 2–5s）mp4，包含触发前后画面。
- **环形保留最近 10 段**：存第 11 段时自动删除最老的一段。每段几 MB，占用极小。
- 训练帧（猫头特写 + 标签）存于独立永久目录，不受 10 段轮换影响。

## 7. 统计

- 每次喝水事件写一条记录（时间戳）到 SQLite。
- 网页显示**今日喝水次数 + 各次时间点**。
- （阶段 3）多天历史趋势。

## 8. 本地网页（在 Mac Mini 上运行，绑定局域网）

- **实时画面**（MJPEG 流）
- **最近 10 段视频**：在线播放 / 下载；每段带 **👍真喝水 / 👎没喝** 反馈按钮
- **今日喝水统计**（次数 + 时间点）
- （阶段 2）隐私时段设置、网页上画水碗区域框、"用反馈重新训练"入口

## 9. 隐私关闭（阶段 2）

- 软件关闭：到达用户设定的隐私时段时，自动停止采集/录制/推流，网页显示"隐私中"。零额外硬件。

## 10. 技术栈

- **Python**：
  - OpenCV — 摄像头采集、帧缓存、写 mp4
  - ultralytics YOLO（如 YOLOv8n/YOLO11n）— 预训练认猫；阶段 2 微调舌头模型
  - FastAPI — 本地网页与 API
  - SQLite — 喝水事件统计、反馈标签

## 11. 模块划分（各自单一职责、可独立测）

| 模块 | 职责 | 依赖 |
|---|---|---|
| `config` | 集中配置：摄像头编号、水碗区域坐标、停留秒数、冷却秒数、片段时长、保留段数、隐私时段等 | — |
| `camera` | 打开 USB 摄像头、持续出帧、维护滚动帧缓存 | OpenCV、`config` |
| `detector` | YOLO 认猫 + 水碗区域 + 停留逻辑 → 产出"喝水事件" | YOLO、`camera`、`config` |
| `recorder` | 收到事件 → 从帧缓存拼 mp4 → 写盘 → 维护"最多 10 段"环形 | OpenCV、`camera`、`config` |
| `stats` | 记录每次喝水事件时间、计算今日次数 | SQLite |
| `feedback` | 存每段视频的用户标注（喝水/没喝），抽训练帧入永久训练集 | SQLite、文件系统 |
| `web` | 本地网页：实时流 / 视频列表与下载 / 今日统计 / 反馈按钮 | FastAPI、上述模块 |
| `trainer`（阶段 2） | 从累积标注微调舌头模型、自动替换当前模型 | YOLO、`feedback` |

## 13. 路线图

- **MVP**：`camera` 采集 + `detector`（YOLO认猫闸门 + 停留触发）+ `recorder`（录片 + 留 10 段）+ `stats`（今日统计）+ `web`（实时画面 / 视频看下载 / 今日统计 / **👍👎反馈按钮，从第一天起攒标注数据**）。先手动运行跑通。
- **阶段 2**：`trainer` 再训练流水线 + 模型自动替换；隐私时段自动关闭；网页画框设水碗区域；做成开机自启服务（launchd，崩溃自动重启）。
- **阶段 3**：多天历史趋势 / 网页美化 / "今日喝水偏少"提醒。

## 14. 验收标准（MVP）

1. 普通 USB 摄像头插上 Mac Mini，程序能读到画面。
2. 猫白天到流动水碗喝水时，能稳定触发并录下一段 2–5 秒 mp4（流动水不造成持续误触发）。
3. 始终只保留最近 10 段视频，超出自动删最老。
4. 本地网页能：看实时画面、列出并播放/下载这 10 段、显示今日喝水次数与时间点。
5. 网页上每段视频可点 👍/👎，标注与训练帧被持久化保存（为阶段 2 训练攒数据）。
6. 全程不向任何外部网络/云端发送画面或视频。

## 15. 已知局限与暂不做（YAGNI）

- 夜间黑灯不检测（无红外，接受漏记夜间）。
- MVP 阶段会把"好奇凑近但没喝"的误计为喝水（靠阶段 2 舌头模型修正）。
- MVP 不做隐私时段自动关闭、不做网页画框设区域、不做开机自启服务（均归阶段 2）。
- 不做内网穿透 / 公网访问（远程靠用户自己远程进 Mac Mini）。
- 不接入任何第三方/云端 AI 识别 API（隐私考虑）。
